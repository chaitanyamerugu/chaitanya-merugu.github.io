Introduction

ECC to S/4HANA migrations are among the highest-risk technical projects an enterprise can undertake. Decades of custom ABAP development — Z-reports, user exits, BAdI implementations, modified standard objects — accumulates into a body of code that may be incompatible with S/4HANA's simplified data model and restricted API set. Without a systematic remediation approach, these incompatibilities surface as runtime errors in production, often at the worst possible time.

This artifact documents the technical remediation framework developed and applied during Pennsylvania's Commonwealth-wide ECC to S/4HANA Private Cloud (AWS-hosted) migration — one of the largest state government SAP migrations in the US. The framework provided a repeatable, risk-minimizing methodology that enabled a zero-disruption go-live for critical HR and Finance operations serving thousands of state employees.

Artifact Description

Objective

To systematically identify, remediate, and validate custom ABAP objects during Pennsylvania's ECC to S/4HANA Private Cloud (AWS-hosted) migration, ensuring zero disruption to critical HR and Finance operations across state-wide systems.

Migration Risks Addressed
  • Custom code using deprecated APIs incompatible with S/4HANA
  • Dictionary object conflicts during SPDD/SPAU adjustment
  • Modified SAP standard objects overwritten by upgrade
  • RAP compatibility gaps in existing ABAP reports
  • Environment deployment sequencing errors (Dev→QA→Prod)
Remediation Framework — Phases
  1. Custom Code Analysis: Used SAP Custom Code Migration Worklist to scan all Z/Y objects. Categorized findings by severity — blocking, non-blocking, and informational.
  2. SPDD Adjustment: Performed Dictionary object adjustment (SPDD) to handle structural conflicts between SAP standard table/structure changes and client modifications.
  3. SPAU Adjustment: Executed Repository object adjustment (SPAU) to re-apply client modifications to upgraded standard ABAP programs, function modules, and includes.
  4. Deprecated API Remediation: Replaced obsolete function modules, BAPIs, and database table accesses with their S/4HANA-compatible equivalents (e.g., MATMAS → Product APIs).
  5. RAP Readiness Validation: Assessed and adapted custom CDS views and OData services for RESTful ABAP Programming Model compliance in the new environment.
  6. Basis Team Collaboration: Coordinated transport sequencing across Development, QA, and Production environments with the Basis team to ensure conflict-free deployment.
  7. Post-Migration Validation: Executed functional regression testing for all HR and Finance modules, verifying payroll runs, leave processing, and financial postings in S/4HANA.

Key Learnings & Takeaways
  • SPDD must be completed before SPAU — dictionary objects underpin repository objects
  • Systematic severity categorization of custom code findings prevents last-minute blocking issues
  • RAP readiness is non-negotiable for any custom development surviving into S/4HANA Cloud
  • A documented remediation framework is reusable across modules and future system upgrades
